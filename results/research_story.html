I tried to teach a machine
what makes a recipe loved.

CHAPTER ONE

Two files that refused to talk

It started simply enough. Epicurious gave me two views of the same world: epi_r.csv — twenty thousand recipes as a huge matrix of binary tags and nutrition numbers — and full_format_recipes.json — the same recipes as raw text, directions and ingredients. I needed them joined. How hard could it be?

My first attempt was the naive one: join them by row position. The result: only 2.8% of rows actually matched. The two files were sorted in completely different orders.

Fine — join on the recipe title, then. That's when I met the second trap: roughly 2,300 titles repeat. There are many distinct recipes all named “Chicken Soup”. A title join would multiply them against each other — a Cartesian explosion of corrupted, cross-matched rows that would have silently poisoned the dataset.

The move that worked: fingerprinting

Both files independently carry five numbers per recipe — rating, calories, protein, fat, sodium. So I built a composite key: title + the five numbers, rounded to three decimals so both files format identically. De-duplicate each side, then a strict, validated 1:1 inner join.

Before doing anything else, I split: 80/20, stratified, random_state=42 — before any feature engineering or fitted statistic. Every transformer in this story is fit on the training split only. That discipline never gets broken, and it ends up mattering.

CHAPTER TWO

The outlier ambush

With clean data in hand, I reached for the default tool: StandardScaler. Then I looked at a sanity panel and stopped cold.

After standard scaling, the test split’s fat column had a standard deviation of 0.034. The entire feature had been flattened into a line. — the diagnostic that changed the preprocessing

The culprit: scraping artifacts. A handful of recipes carried impossible calorie and sodium values, and those few monsters absorbed virtually all the variance. Any distance-based or L2-regularised model downstream would have been crippled — before a single hyperparameter was ever tuned.

StandardScaler

Mean/std are exquisitely sensitive to outliers.

fat std ≈ 0.034

RobustScaler

Median center, IQR scale — outliers can't vote.

fat IQR ≈ 0.962

There was a quiet elegance available here too: pairing SimpleImputer(strategy="median") with RobustScaler means every imputed value (about 21% of nutrition cells were missing) lands at exactly 0.0 in scaled space — the perfectly neutral “I know nothing” position.

Lesson #1 was already on the books: distributional pathology in the inputs outranks model choice. Diagnose the space before you tune anything that lives in it.

CHAPTER THREE

Crafting the culinary features — and designing a fair fight

The CSV's 674 tags say what kind of recipe something is. But the raw text knows things the tags don't: how elaborate a recipe is, whether it sears or simmers, whether it asks you for saffron or sriracha. I built a custom transformer, CulinaryFeatureExtractor, to mine nine features from the directions and ingredients:

The nine engineered features

3 numerics — num_steps, num_ingredients, avg_words_per_step (robust-scaled like the nutrition).

6 keyword binaries — curated domain-knowledge groups: high-heat techniques (sear, broil, char…), low-and-slow (braise, confit, smoke…), technical execution (emulsify, deglaze, julienne…), prep & patience (marinate, brine, overnight…), flavor development (caramelize, reduce, roux…), premium ingredients (truffle, saffron, miso…).

Then the experimental design — the part I'm still proud of: two parallel matrices. Baseline (678 columns: nutrition + tags) and Advanced (687: the same + my nine features). Identical rows, identical split. Whatever difference shows up downstream is attributable to those nine columns alone. Every model would run on both — a controlled A/B test of my own feature engineering.

Seven models were lined up: Perceptron as the sanity floor, Logistic Regression as the interpretable workhorse, AdaBoost, PCA+KNN, and — added later for reasons you'll see — Random Forest and an MLP. Each got its own training script writing to its own results/<slug>/metrics.json; a pure-read aggregator assembles the verdict. One constant, 42, bound in exactly one file, seeds everything.

CHAPTER FOUR

The solver that lied politely

Logistic Regression trained fine — or so it seemed. There was a convergence warning from the default lbfgs solver, but accuracy looked stable, F1 looked stable, and warnings are easy to dismiss as cosmetic. I raised max_iter and moved on. Almost.

While inspecting per-recipe probabilities, one test recipe caught my eye — Lamb Köfte with Tarator Sauce:

# same model, same data — only the iteration budget differs max_iter = 2000 → p(Hit) = 0.9997 max_iter = 5000 → p(Hit) = 0.0000

A 99-percentage-point swing on a single prediction. The optimizer had never been anywhere near the optimum; the aggregate metrics had simply absorbed the chaos and smiled. Switching to liblinear — coordinate descent, at home on sparse high-dimensional binary data — made convergence clean and every probability reproducible.

Stable summary metrics do not imply a converged model. Convergence is a correctness property, not a performance knob. — Lesson #2
CHAPTER FIVE

The silent bug — six features that were ghosts

By now the fleet was training, numbers were flowing, and a story was writing itself: the linear model assigned my six keyword features coefficients of zero, and I had a beautiful explanation ready — L2 regularisation shrinking collinear features, the keyword groups being redundant with the existing tags. Plausible. Educated. Wrong.

Then an audit asked one question my pipeline had never asked itself:

X_train_advanced.filter(like="has_").sum()
has_high_heat_techniques 0 has_low_and_slow_techniques 0 has_technical_execution 0 has_prep_and_patience 0 has_flavor_development 0 has_premium_ingredients 0

All six columns were zeros. In every model. In every run. From the beginning.

The curated keyword lists existed — but only inside a sanity-check function nobody called. The pipeline itself constructed the extractor bare, with empty keyword groups, and the extractor's schema-stability contract did exactly what it was designed to do: emit all-zero columns without a word of complaint. No crash. Plausible shapes. Reasonable metrics. An entire interpretive story written about features that did not exist.

The fix — and the vow

1. The domain knowledge moved to one constant, CULINARY_KEYWORDS, consumed by a factory: make_culinary_extractor().

2. The pipeline now uses that factory — the words actually reach the model.

3. And a guard: the build raises an error if any engineered binary is all-zero on the training split. This class of bug doesn't get to be silent twice.

With the features alive, they fired on 14.6%–39.0% of training recipes — healthy, informative rates. Everything had to be retrained, and every number that follows comes from the fixed pipeline.

A feature pipeline should fail loud when a feature is degenerate — models will happily train around dead inputs and report nothing. — Lesson #7, the expensive one

Random Forest took the crown — 0.6239 accuracy, 0.6753 F1, a clear +2.28pp / +3.49pp over the converged Logistic Regression. And the engineered features? Vindicated where it counts: every model that can exploit interactions gained from them (RF +1.98pp, MLP +1.54pp, improved KNN +1.29pp accuracy), while the purely linear models stayed flat. The signal was real — it just wasn't linear.

And the most satisfying twist: Random Forest's feature-importance ranking inverted the linear picture. The continuous numerics that L2 had buried — avg_words_per_step (rank 597/687 in LR!) — sat at the very top of RF's ranking, with all four nutrition columns and my other two numerics right behind, and all six keyword features inside RF's top 20, above ~670 of the editorial tags.

Feature utility is not a property of the feature. It is a property of the (feature × model class) pair. — Lesson #3, the project's deepest finding
CHAPTER SEVEN

Two rescue missions

Rescue #1 — the network that memorised everything.

The MLP (128, 64) — about 100,000 parameters against 14,578 recipes — did exactly what unsupervised capacity does: it drove its training loss to 0.0241, essentially memorising the training set, while its test accuracy crashed to 0.5728, far below plain Logistic Regression. More power, worse model.

The cure cost nothing: validation-based early stopping. Hold out 15% of the training split internally, watch validation accuracy, stop after 10 patient epochs, restore the best weights (epoch 7 of 18). Same architecture, same seed, same data:

No early stopping

53 epochs · loss → 0.0241

Acc 0.5728 · F1 0.5936

Early stopping

18 epochs · restored @ 7

Acc 0.6126 · F1 0.6632

+3.98pp accuracy, +6.96pp F1 — and the regularised network now clears the linear ceiling instead of collapsing under it. One stopping rule is the entire difference between “worse than linear” and “better than linear.”

Rescue #2 — the projection that ate 686 columns.

PCA(0.90) + KNN scored barely above coin-flipping, and the tempting explanation was theoretical: “KNN can't do Euclidean distance in sparse binary spaces.” True — but not what happened. One structural diagnostic told the real story:

pca.n_components_ → 1 # one. out of 687.

Even after robust scaling, residual outlier variance in the four nutrition columns swallowed the entire 90% variance budget. PCA dutifully kept a single nutrition-aligned axis and discarded everything else; KNN was doing similarity search on a one-dimensional shadow of the data. The fix — drop the four nutrition columns before projecting — exploded the retained space to 203 / 178 components and bought back +4.5 to +6.3pp of accuracy. Partial, honest, diagnostic-driven.

CHAPTER EIGHT

Reading the model's mind — success, disaster, and a trap

With a converged Logistic Regression, the coefficients finally meant something. The strongest Hit signals read like a party invitation: kentucky derby, pasadena, brisket, sangria, kahlúa, sukkot — occasion recipes, rated generously by people already in a good mood. The strongest Miss signals read like obligations: leftovers, whole wheat, slow cooker — the “I'd rather have the real thing” effect. (And some honest leakage: tags like #cakeweek and pittsburgh are editorial metadata, not cooking.)

The single most reliable engineered signal stayed simple: num_ingredients, coefficient +0.40 — holding the tags constant, more ambitious recipes rate better.

Last act: tuning the decision threshold. And here the process caught itself making a methodological mistake — an earlier draft picked the threshold by sweeping the test set, which is quiet self-deception. The redo: select on a 15% validation split carved from training, then evaluate once on test.

What the validation-selected thresholds did on test

t = 0.30 (max-F1): F1 0.7097 — but it labels 90.5% of recipes “Hit” (true rate 53.4%), FP/FN balance 19.4. The “best F1” threshold is a trap that the harmonic mean hides.

t = 0.55 (most balanced): F1 0.5989, accuracy 0.6005, balance 0.80 — nearly symmetric errors at virtually no accuracy cost. The recommended operating point.

Accuracy barely moves across t = 0.40–0.55. What moves is which class absorbs the errors — a choice that belongs to the use case, not the optimiser.

And the seven lessons, the actual takeaway of the project:

1 · Outliers in the input space beat model choice

No hyperparameter could have rescued a feature space flattened to std 0.034. Diagnose distributions before tuning models.

2 · Convergence is correctness, not performance

Aggregate metrics stayed stable while individual probabilities swung 99 points. A warning is data.

3 · Feature utility is model-dependent

The same features ranked 597/687 in LR and #1 in RF. Never discard a feature on one model class's testimony.

4 · F1 alone is unsafe — and select thresholds on validation

The max-F1 threshold called 90.5% of everything a Hit. Check balance, and never let the test set pick your operating point.

5 · Read structural diagnostics before theorising

“KNN struggles in sparse spaces” was true and irrelevant; pca.n_components_ == 1 was the actual story.

6 · Capacity is a liability without regularisation

The same MLP sits 2.8pp below or 1.2pp above the linear baseline depending solely on the stopping rule.

7 · Audit that engineered features actually reach the model

Six features shipped as zeros, silently, while a plausible story was written about them. Fail loud on degenerate inputs.
